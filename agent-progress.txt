[database] Build database access utilities (lib/db.ts)  — passes=true
  - Created lib/db.ts: shared SQLite handle + typed helpers.
  - Singleton getDb() (lazy open, cached) with WAL + foreign_keys pragmas.
  - query<T>(sql, params) -> rows; get<T>(sql, params) -> row|null;
    run(sql, params) -> { changes, lastInsertRowid } (normalized to numbers).
  - Prepared-statement cache; all SQL parameterized via $name binds.
  - closeDb() for test isolation.
  - DESIGN DECISION (spec-sanctioned, SPEC §3.2/§17): thin driver abstraction.
    Uses bun:sqlite under Bun (dev/build/start run with --bun) and falls back to
    node:sqlite (DatabaseSync) under plain Node, so the data layer is testable
    under Vitest (which runs on Node) without breaking production. Driver picked
    at runtime via createRequire + a variable module specifier (bundler-safe).
  - Verified: dual-runtime smoke test (Bun + Node, all asserts pass), tsc via
    `bun run build` (OK), `bun run lint` (clean).

[database] Create notes table with indexes  — passes=true
  - Added applyNotesSchema() to lib/db.ts, called from openDatabase() right after
    the pragmas, so the `notes` table + indexes are created idempotently on cold
    start (CREATE TABLE/INDEX IF NOT EXISTS). This is per CLAUDE.md's decision
    ("notes schema lives in lib/db.ts"); supersedes the earlier "db.ts creates no
    tables" stance from the lib/db.ts iteration.
  - Schema (SPEC §5.2 shape, prd column names): id TEXT PRIMARY KEY NOT NULL,
    user_id TEXT NOT NULL REFERENCES "user"(id) ON DELETE CASCADE, title TEXT,
    content_json TEXT NOT NULL, is_public INTEGER NOT NULL DEFAULT 0,
    public_slug TEXT UNIQUE, created_at/updated_at TEXT NOT NULL.
    Indexes: idx_notes_user_id, idx_notes_is_public; public_slug is indexed by
    its UNIQUE autoindex (the share-link lookup path) — no separate named index.
  - NAMING DECISION: used prd's content_json / public_slug (not SPEC §5.2's
    content / public_id). The prd defines `passes` and is internally consistent
    across the data-layer tasks (Note type uses camelCase contentJson/publicSlug,
    and there's a getNoteByPublicSlug task). TS Note type (next task) will map
    snake_case rows -> camelCase. Flagging so a future iter doesn't "fix" this.
  - FORWARD-FK / ORDERING (corrects the note below about user existing first):
    by design getDb() (-> openDatabase -> applyNotesSchema) runs BEFORE auth's
    migrations create `user` (lib/auth.ts calls getDb() then runs migrations).
    SQLite resolves REFERENCES at write time, not DDL time, so creating `notes`
    with a forward FK to a not-yet-existing `user` is fine; the FK becomes
    enforceable once auth creates `user`. Verified end-to-end under bun:sqlite
    AND node:sqlite with the real better-auth migration path.
  - VERIFIED: 17-check throwaway bun script (all 8 cols + types/NOT NULL,
    is_public DEFAULT 0, FK enforced on insert, ON DELETE CASCADE, UNIQUE rejects
    dup public_slug + allows many NULLs, all 3 index targets present, id NOT NULL
    rejects NULL). Adversarial 3-lens review (compliance/robustness/invariants)
    all voted "ship", nits only. `bun run build` PASS, `bun run lint` PASS.

    deep-frozen so the shared constant can't be mutated — clone before editing.
